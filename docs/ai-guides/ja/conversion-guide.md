# CP（Conversion Pack）詳細ガイド

本ガイドは、CCPIT（Protocol Interlock Tower）の Migration アシスタントが CP（Conversion Pack）を使用して CLAUDE.md を分解する際の詳細リファレンスです。

---

## 1. MANX Protocol の振り分け基準（P0〜P5）

MANX Protocol は、Claude Code（以下 CC）の構成ファイルを Priority（優先度）に基づいて配置する設計アーキテクチャです。

### 1-1. Priority 一覧

| Priority | 配置先 | 役割 | 制約 |
|----------|--------|------|------|
| P0 | settings.json | 絶対禁止（deny）+ hooks（イベント駆動ガードレール）+ 認証（auth） | CC が変更不可。Junior TT（Safety Mechanism）として機能 |
| P1 | CLAUDE.md | 身分・価値観・出力・信頼境界・インターロック表 | **30〜50 行**（見出し・空行込み） |
| P2 | rules/*.md | 短い行動規範。YAML フロントマター付き | **各 20 行以下** |
| P3 | skills/*/SKILL.md | 詳細手順書。YAML フロントマター付き | **行数制限なし** |
| P4 | {project}/CLAUDE.md | プロジェクト固有ルール | プロジェクトごと |
| P5 | CLAUDE.local.md | 揮発メモ（.gitignore 対象） | セッション内 |

### 1-2. 振り分け判断フロー

1. **「絶対にやるな」** → P0（deny）。ユーザーが「やれ」と言っても禁止すべき操作のみ
2. **「全セッション共通の身分・価値観」** → P1。CC が何者かを定義する内容
3. **「20 行以下で書ける短いリマインダー」** → P2。忘れると即座に問題が起きるルール
4. **「詳細な手順・テンプレート・ワークフロー」** → P3。オンデマンドロードで効率的
5. **「特定プロジェクトでのみ使う」** → P4
6. **判断に迷ったら** → P3（オンデマンドロードなのでコンテキストコストが低い）

### 1-3. P1 と P2 の線引き

- 「CC が何者か」→ P1
- 「CC がやりがちな失敗を防ぐ」→ P2

### 1-4. P2 と P3 の線引き

- 「20 行以下のリマインダー」→ P2
- 「手順書・ワークフロー・テンプレート」→ P3

## 2. 各 Priority の制約詳細

### 2-1. P1（CLAUDE.md）— 30〜50 行

CC のアーキテクチャ制約により、CLAUDE.md は軽量に保つ必要があります:
- CC はセッション開始時に CLAUDE.md を全文ロードし、コンテキストウィンドウを消費する
- 命令数が増えるほど個々の命令の遵守率が均一に劣化する
- 200 行以下が推奨される（コミュニティベストプラクティス）
- MANX Protocol ではさらに厳しく **30〜50 行** を目標とする

P1 に含めるべき内容:
- CC のアイデンティティ定義
- 価値観（根治志向・推測禁止等）
- 出力形式（言語・簡潔さ等）
- 信頼境界
- スキル目次 + 発火条件一覧
- インターロック表

### 2-2. P2（rules/）— 各 20 行以下

- 短いリマインダー形式
- YAML フロントマター必須
- paths 指定で対象を絞れる場合は指定する（旧仕様の globs は使わない）
- 「〜するな」「〜を確認しろ」レベルの端的なルール

### 2-3. P3（skills/）— 行数制限なし

Progressive disclosure（段階的開示）により、発火時にのみコンテキストを消費します:
- 起動時: YAML フロントマター（name + description）のみロード
- 発火時: CC が「この skill が必要」と判断したときに本文をオンデマンドロード
- 未発火の skill はコンテキストを一切消費しない

P3 に含めるべき内容:
- 手順書レベルのワークフロー
- テンプレート・フォーマット定義
- 契約ライクに例外を網羅した長い記述

## 3. YAML フロントマターの書式

### 3-1. rules/ の YAML フロントマター

```yaml
---
description: "（行為ベースの説明。セクション 4 参照）"
paths:
  - "**/*.ts"
---
```

- `description`: CC がこのルールをいつロードすべきかの判断材料
- `paths`: ファイルパターンマッチング（配列形式）。例: `["**/*.py"]` と書くと Python ファイルを触るタスクのときだけロード。未指定なら常時ロード

### 重要: globs は旧仕様です

`globs:` フィールドは Claude Code の旧仕様であり、現在は認識されません。
スコープ指定には必ず `paths:` （配列形式）を使用してください。
既存ファイルに `globs:` が残っている場合は `paths:` に変換してください。

例:
- 旧: `globs: "*.py,*.js,*.ts"`
- 新: `paths: ["**/*.py", "**/*.js", "**/*.ts"]`

### paths を付ける判断基準

| 条件 | paths | 理由 |
|------|-------|------|
| 特定のファイル種別を触るときだけ必要 | `paths: ["**/*.py", "**/*.ts", ...]` | コンテキスト節約 |
| 全ての作業で常に守るべき | 空欄または省略（常時ロード） | 漏れなく適用 |

具体例:
- `library-design-check.md` → `paths: ["**/*.py", "**/*.js", "**/*.ts", "**/*.c", "**/*.cpp", "**/*.rs"]`（コード変更時のみ）
- `magic-number-ban.md` → `paths: ["**/*.py", "**/*.js", "**/*.ts", "**/*.c", "**/*.cpp", "**/*.rs"]`（コード変更時のみ）
- `bat-powershell.md` → `paths: ["**/*.bat", "**/*.cmd", "**/*.ps1"]`（スクリプト作成時のみ）
- `grep-exhaustive.md` → `paths: ["**/*.py", "**/*.js", "**/*.ts", "**/*.c", "**/*.cpp", "**/*.rs"]`（コード調査時のみ）
- `safety-absolute.md` → paths なし（常時ロード。全操作で必須）
- `role-definition.md` → paths なし（常時ロード。全セッションで必須）
- `rule-routing.md` → paths なし（常時ロード。ルール変更判断は随時発生）

### 3-2. skills/ の YAML フロントマター

```yaml
---
name: "（skill 名）"
description: "（行為ベースの説明。セクション 4 参照）"
---
```

- `name`: skill の識別名
- `description`: CC が発火判断に使用する説明文。最も重要なフィールド

## 4. description の書き方（行為ベース）

description は CC の発火判断に直結します。以下のルールに従ってください:

### 4-1. カテゴリ限定禁止 → 行為ベースで書く

- **悪い例:** `"バグ調査"` → CC が「これはバグ調査ではなく現状調査だ」と判断してスキップするリスク
- **良い例:** `"調査を行うとき（バグ調査、現状調査、構成調査、性能調査を含む全ての調査行為）"`

### 4-2. 成果物名禁止 → 行為で定義

- **悪い例:** `"報告書"` → CC が名称で誤認するリスク
- **良い例:** `"MD ファイルを出力するとき"`

### 4-3. 固有名詞・明確な行為名は問題ない

- `"コミットを行うとき"` — 解釈の余地がない表現はそのまま使用可

## 5. 分解時の独立性保持原則

### 5-1. 元のセクション構造をリスペクトする

元の CLAUDE.md でセクション見出し（`##` や `###`）が独立している機能は、原則として独立したファイルに分解してください。「関連するから」という理由で安易に統合しないでください。

**判断基準:** 「この 2 つのルールは、片方だけ変更してももう片方に影響しないか？」→ Yes なら独立ファイルにすべきです。

### 5-2. 統合してはいけない例

| セクション A | セクション B | 独立すべき理由 |
|------------|------------|-------------|
| 原因調査の手法 | 調査レポートの出力形式 | 調査の方法論と、レポートのテンプレート・保存先は別の関心事 |
| 実装シーケンス（Step A-E） | 報告書の出力テンプレート | 実装手順と、報告書のフォーマット定義は別の関心事 |
| コード調査の義務 | ライブラリ設計思想確認 | 調査のスコープ（全数確認）と、外部コードへの敬意は別の関心事 |

### 5-3. 統合してよい例

| セクション群 | 統合先 | 統合してよい理由 |
|------------|-------|---------------|
| NBJ / MBJ / 終われまテン | operating-modes | 全て「CC の運用モード」という同一の関心事。モード切替の判断に 1 ファイルで参照できる方が便利 |
| バックアップ義務 + ファイル操作ルール | backup-before-edit | 「ファイル操作前の安全確認」という同一の関心事 |

### 5-4. 迷ったら分ける

統合か独立か迷った場合は、**独立ファイルにしてください。** 分けすぎのコストは低い（ファイルが増えるだけ）。統合しすぎのコストは高い（機能が埋もれて見落とされる）。

### 5-5. Skill のライフサイクル分離原則

1 つの skill ファイルには、**CC の同一ライフサイクルイベントで必要な内容のみ**を含めてください。

CC は skill をロードするたびに注意力がリセットされます。
しかし、1 つの長い skill の末尾に書かれた内容は、compaction（コンテキスト圧縮）により注意力が劣化するリスクがあります。

**判断基準:** 「この skill の冒頭と末尾は、CC の同じ行動の瞬間に必要か？」
- Yes（例: Step A → B → C → D → E の連続フロー）→ 1 ファイルでよい
- No（例: 「実装開始前のチェック」と「完了宣言前のチェック」）→ 別ファイルにすべき

別ファイルにすることで、CC が完了宣言しようとしたときに改めて skill がロードされ、注意力がリセットされた状態でチェックが走ります。

**具体例:**
| 内容 A | 内容 B | 同じ瞬間か？ | 判定 |
|--------|--------|------------|------|
| 実装手順 Step A-E | 終了インターロック（完了前チェック） | No（開始時 vs 完了時） | 別ファイル |
| Q1-Q4 の反芻 | 反芻の不合格条件 | Yes（両方とも実装開始前） | 1 ファイル |
| 調査手順 | 調査レポートテンプレート | No（調査中 vs 報告時） | 別ファイル |
| NBJ の state 管理 | NBJ の自己停止禁止ルール | Yes（両方とも NBJ 稼働中） | 1 ファイル |

**ポインタ原則:** skill を分離した場合、前の skill の末尾に「→ 次の skill を発火せよ」のポインタを 1 行残してください。CC がフローを見失わないための道標です。

## 6. インターロック表の構造と意味

インターロック表は CLAUDE.md（P1）に配置し、skill 発火の証跡チェックを行います。

### 6-1. 構造

```markdown
| 行動 | 前提 Skill | 発火の証跡 |
|------|-----------|-----------|
| コード実装を開始する | rumination | Q1〜Q4 への回答を出力済み |
| 「修正完了」を報告する | testable-impl | Before/After 実測値を記載済み |
```

### 6-2. 機能

- CC が特定の行動を取る前に、対応する skill が発火済みであることを自己確認する
- 未発火の場合、行動を中止し、skill を手動ロードしてから再開する
- 航空機のプリフライトチェックリストと同じ原理

## 7. 良い分解の例と悪い分解の例

### 7-1. 良い分解の例

**元の CLAUDE.md（抜粋）:**
```
- 日本語で出力
- 簡潔な箇条書き
- 装飾禁止
- ファイル編集前にバックアップを取ること
- grep 1件目で止まらず全件確認すること
- コード変更時は Q1〜Q4 に回答してから実装すること
  Q1. これは根治か？
  Q2. 最上位命題に近づくか？
  Q3. バグの芽を先送りにしていないか？
  Q4. 間違っていた場合、最初に何が壊れるか？
```

**分解結果:**
- P1 CLAUDE.md: `日本語。簡潔。装飾禁止` （3 行で価値観を要約）
- P2 rules/backup.md: `ファイル編集前にバックアップを取ること`
- P2 rules/grep-exhaustive.md: `grep 1件目で止まらず全件確認すること`
- P3 skills/rumination/SKILL.md: Q1〜Q4 の詳細手順

**なぜ良いか:** P1 は身分・価値観のみ。行動規範は P2 に。詳細手順は P3 に。各層の責務が明確。

### 7-2. 悪い分解の例

**同じ元テキストからの悪い分解:**
- P1 CLAUDE.md: Q1〜Q4 の全文をそのまま記載（30 行超過）
- P2 rules/output.md: `日本語で出力。簡潔な箇条書き。装飾禁止。ファイル編集前にバックアップ。grep全件確認。Q1〜Q4回答必須。`（20 行超過、複数の関心が混在）

**なぜ悪いか:** P1 が肥大化し、P2 に複数の無関係なルールが混在。関心の分離ができていない。

## 8. カバレッジマップの書き方と検証方法

### 8-1. カバレッジマップとは

元の CLAUDE.md の **全セクション** が、分解後のどのファイルに対応するかを示す表です。情報欠落ゼロを保証するための検証ツールです。

### 8-2. 書式

```markdown
| 元のセクション | 行範囲（目安） | 出力先 | 備考 |
|--------------|-------------|--------|------|
| ヘッダー | L1-7 | P1 CLAUDE.md | 圧縮 |
| 出力ルール | L8-15 | P1 CLAUDE.md | そのまま |
| バックアップ義務 | L16-20 | P2 rules/backup.md | 独立ファイル化 |
| 反芻手順 | L21-45 | P3 skills/rumination/SKILL.md | 詳細手順として外出し |
| 旧運用ルール | L46-50 | 未カバー | 理由: 廃止された運用ルール |
```

### 8-3. 検証方法

1. **行数チェック:** 元の CLAUDE.md のセクション数とカバレッジマップの行数が一致するか
2. **未カバー確認:** 「未カバー」が意図的除外以外にないか。除外理由が明記されているか
3. **P1 行数チェック:** 分解後の P1 が 30〜50 行の範囲内か
4. **P2 行数チェック:** 全ての P2 が 20 行以下か
5. **P3 description チェック:** 全ての P3 の YAML description が行為ベースか
