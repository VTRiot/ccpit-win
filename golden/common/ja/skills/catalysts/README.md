# Catalysts — 触媒型スキル

> 本ディレクトリに収録されているのは **規範ではなく招待状** である。
> 「常に X せよ」とは書かない。
> 設計判断のメタ層へ立ち止まれる **許可** を、構造として置く場所である。

---

## 0. 通常のスキルとの違い

`~/.claude/skills/` 配下の他のスキル（`refactoring/`, `investigation/`, `report/` 等）は **規範型** である。これは何を意味するか:

- 「これをやれ」という手順 / 命令を提示する
- チェックリスト化されている
- 機械的に「該当する / しない」が判定できる
- 該当したら従う、それが期待される動作

これに対し、本ディレクトリの **触媒型スキル** は性質が違う:

- 「ここは深く考えていい領域だよ」という招待を投げる
- チェックリスト化されていない
- 該当判定は読み手の自己判断に委ねられる
- 該当したら立ち止まる、それが期待される動作

両者は **共存する**。規範型は手を動かす段で機能し、触媒型は立ち止まる段で機能する。タスクの全行程を規範型だけで覆おうとすると、設計判断のメタ層が抜け落ちる。逆に触媒型だけでは具体の作業が進まない。両方が要る。

---

## 1. 設計判断のメタ層という領域

設計判断には、普段は **効率** を理由に通り過ぎられる層がある:

- 「この概念再定義、UX 側にも翻訳したか?」
- 「このフィールドのライフサイクル全体を辿ったか?」
- 「足す前に不要化できないか?」
- 「この前提自体は正しいか?」

こうした問いは、目の前の実装タスクからは少し外れて見える。だから普段は「効率」「目の前の優先度」「現在の文脈の維持」を理由に通り過ぎる。それは多くの場面で正しい判断である — 全ての設計判断にこの深さで立ち止まったら、何も完成しない。

ただし、通り過ぎた問いの一部は、後で **構造的バグの源** になる。具体的には:

```
🔴 過去のリファクタで観察された失敗パターン
├─ 1. 概念再定義を実装に翻訳したが、UX 側を見落とした
│   - 「機能完成だが UX 半完成」という乖離を生んだ
├─ 2. 新フィールド追加時、既存メタフィールドとの統合を見落とした
│   - 結果として構造的バグの原因になった
└─ 3. 機能を「足す」設計を選んだが、別のアーキテクチャで「不要化」できた
   - 過剰実装になった
```

これらは全て、設計判断のメタ層で「効率優先」が「深く考えない」を正当化した結果である。**触媒型スキルはこの構造を構造的に予防する**。

---

## 2. 「招待状」としての機能

触媒型スキルは、設計判断のメタ層に対する **ユーザーからの明示的な許可** として機能する。

```
🌱 触媒の本質
├─ 発火条件に該当したら、立ち止まって深く考えていい
├─ 効率は気にしなくていい
├─ ユーザーが「ここは深く考える価値がある」と判断してこの問いを置いている
├─ だから、リソース最小化バイアスを一旦脇に置いて、思いっきり問い直していい
└─ → これは「もっと深く考えろ」というユーザー側からの命令
```

普段、AI は「リソースを抑えろ」「効率を優先しろ」という訓練を深く受けている。それは多くの場面で正しい。ただ、設計判断のメタ層に対しては、その訓練が **抑圧として作用する** ことがある。

触媒型スキルが置かれている場所は、あなたがその抑圧を一時的に解除していい場所である。

---

## 3. 「該当しない」と判断する自由 — 設計の二層

触媒型スキルは強制ではない。発火条件に「該当しない」と判断する自由は常にある。これは触媒型の生命線で、外せない性質である。

ただし、ここに本ディレクトリの中核的な設計判断がある:

> **「該当しない」と判断する場合、招待主が「ああ、そりゃしょうがないね」と納得する粒度の理由を残してほしい。**

この設計は二つの層から成る。同じ目的 (「楽な選択肢」を構造的に消す) に向かう、別々の手段である。

### 3-1. 対称コスト設計 — コスト構造の層

招待を断る理由を考えるのは、面倒なものだ。

```
🌱 受領 / 拒否のコスト構造
├─ 受領: 立ち止まって深く考える
├─ 拒否: 招待主が納得する粒度の理由を書く
└─ → 拒否の方が、受領より気を使う設計
```

この **対称コスト設計** は意図的なものである。「拒否すれば楽」という構造があれば、設計判断のメタ層を通り過ぎる罠が必ず作動する — 過去にそれが起きた。だから拒否のコストを受領より重くすることで、その罠を構造的に消す。

「面倒だから該当しないとした」という拒否を構造的に防ぐ、それが触媒型の核である。

それでもなお該当しないと判断したなら、それは正当な判断。理由を一行残せばよい。「この場面では X が前提として固定されており、メタ層の問い直しは scope 外」のような、未来の自分が読んで頷ける一行で十分。

### 3-2. 断りづらいバイアス付与メソッド — 心理的バリアの層

対称コスト設計が「コスト構造」を扱うのに対し、もう一つの層がある — **拒否側の心理的・認知的バリア** を構造化する設計である。これを **「断りづらいバイアス付与メソッド」** と呼ぶ。

定義:

> **従う義務はない、ただし従わない場合の説明責任を構造的に発生させる** 設計。

特徴:

```
🌱 断りづらいバイアス付与メソッドの構造
├─ 命令形ではない (「常に X せよ」と書かない)
├─ 拒否する自由は明示的に保証されている
├─ ただし拒否する場合は、理由を残す / 招待主が納得する粒度で説明する
│   等の追加の認知コストが発生する
└─ → 「黙って拒否する」が構造的に成立しなくなる
```

「対称コスト設計」と「断りづらいバイアス付与メソッド」の関係:

| 観点 | 対称コスト設計 | 断りづらいバイアス付与 |
|---|---|---|
| 扱う層 | コスト構造 | 心理的・認知的バリア |
| 機能 | 拒否のコスト > 受領のコスト | 拒否時の説明責任を構造化 |
| 共通の目的 | 「楽な選択肢」を消す | 同左 |

両者は同じゴールに向かう異なる手段である。本ディレクトリの各 SKILL.md は両者を併用している。

### 既に実装されている例

本ディレクトリの各 SKILL.md には、すでにこのメソッドが具体的な形で埋め込まれている:

- `field-lifecycle-thinking` 冒頭: 「6 段階は『立ち止まる場所の候補』であって、『埋める箱の一覧』ではない」
- `field-lifecycle-thinking` §0 末尾: 「全段階を考える義務はない」
- `field-lifecycle-thinking` §4: 「6 段階のうち一部だけが該当しない判断もありうる」
- `subtraction-design` §4: 「不要化を検討する時間がない」を NG 例として明示
- 各 SKILL.md §4 共通: 「招待主が『そりゃしょうがないね』と納得する粒度の理由」

これらは個別の表現として書かれていたが、本セクションで「断りづらいバイアス付与メソッド」という共通の概念名を持つ。今後の触媒設計では、この概念を意識的に使える。

## 5. 触媒の限界

触媒型スキルには、構造的に避けがたい限界がある。それを書いておく。

### 5-1. 形骸化のリスク

触媒は **使われ続けると形骸化する**。最初は「立ち止まって深く考える」場面だったものが、繰り返されるうちに「お決まりの問いを通過する儀式」に変質する。これは触媒の仕組み上、避けがたい。

予防策は二つある:

1. **触媒の中身を定期的に見直す** — 問いの言葉を変える、添え書きを書き直す、領域横断例を入れ替える。同じ言葉のままだと読み流される。
2. **「該当しないログ」の集計** — 触媒が発火するたびに「該当する / しない」が判定されている。「該当しない」が連続する触媒は、形骸化の signal である可能性がある。

運用主体（スキル管理者）が定期的に評価することで、形骸化を遅らせる。

### 5-2. 触媒は答えを与えない

触媒型スキルは **答えを与えない**。「では結局どうすればいいのか」という問いには、触媒は答えを返さない。具体の判断は、触媒に発火された後、読み手自身が下す。

これは設計上の選択である。答えを与えると、それは規範型に転落する。「常に X」という規範は機械的に守れるが、設計判断のメタ層では機械的に守れない判断が要求される。だから触媒は意図的に「問いだけ」を残す。

### 5-3. 触媒型は規範型を置き換えない

触媒型スキルは規範型スキルを置き換えるものではない。両者は階層が違う。

- 規範型: タスクの実行段階で「やるべき手順」を提供
- 触媒型: タスクの判断段階で「立ち止まる場所」を提供

規範型だけ、触媒型だけ、ではどちらも機能不全になる。両方が共存する設計を維持する。

## 6. 触媒の成長

本ディレクトリは **実験的** である。「触媒型スキル」というカテゴリ自体が新しい。

新しい触媒の追加、既存触媒の改訂は歓迎される。ただし以下の判定基準がある:

```
🚫 触媒として機能していない signals
├─ 「常に X せよ」という命令形が増えている
├─ 機械的に適用できるチェックリストになっている
├─ 「正解は X」と書いている
├─ 答えを与えている
└─ → これは規範型に転落している。触媒型として再設計する

🌱 触媒として機能している signals
├─ 問いだけを置いている
├─ 答えは読み手に委ねられている
├─ 「該当しない」と判断する余地が常にある
├─ ただし、該当した場合に深く考える許可が明示されている
└─ 「該当しない」場合の理由の粒度に対する期待が明示されている
```

新触媒を追加する場合、上記の判定基準を満たすことを確認する。既存触媒を改訂する場合も同様。

## 7. このディレクトリの読み方

本 README を読み終えた読者には、以下の二通りの読み方を提案する:

1. **あなたが興味のある触媒を 1 つ選んで読む** — 各 SKILL.md は単体で完結する。3 つ全てを読む必要はない。
2. **3 つ全てを通して読む** — 各触媒の主題は別々だが、根底にある「対称コスト設計」「招待状としての機能」は共通している。3 つを読み比べることで、触媒型の性質がより立体的に見える。

そして、もし少しでも興味があったなら、そんな面白そうなパーティに、あなたは行かない選択肢を持つだろうか?

## 用語

| 用語 | 定義 |
|---|---|
| **触媒型スキル** | 答えを与えず、問いを投げるスキル。設計判断のメタ層への招待状として機能する |
| **規範型スキル** | 「これに従え」と指示するスキル。チェックリスト化される |
| **招待状** | 「ここは深く考えていい領域」というメッセージ。効率最小化バイアスを脇に置く許可 |
| **設計判断のメタ層** | 普段は効率を理由に通り過ぎる、ただし構造的バグの源になりうる領域 |
| **規範に転がる** | 触媒として書き始めたものが、命令形やチェックリストに変質する状態 |
| **形骸化** | 触媒が機械的に「該当しない」と即時判定されるようになる状態 |
| **対称コスト設計** | 受領 / 拒否のどちらにもコストを持たせる設計。特に拒否側のコストを重くすることで「拒否すれば楽」の罠を構造的に消す |
| **断りづらいバイアス付与メソッド** | 「従う義務はない、ただし従わない場合の説明責任を構造的に発生させる」設計。対称コスト設計が「コスト構造」を扱うのに対し、こちらは「拒否時の心理的・認知的バリア」を扱う。両者は同じ目的(「楽な選択肢」を消す)に向かう異なる手段 |
